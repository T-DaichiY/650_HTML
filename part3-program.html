PART III Overview — Program / プログラミング
①
💬 Code is communication / コードは「コンピュータへの命令」だけではない
PART IIIの中心メッセージは、コードはコミュニケーションの道具だということです。もちろんコードはコンピュータに命令します。しかし同時に、他の人、そして未来の自分に「なぜこの分析をこう進めたのか」を伝えます。だから、速く書けるコードよりも、あとで読んで意味がわかるコードの方が長期的には強いです。
考え方
今の自分 → 未来の自分へのメモ
数週間後の自分は、ほぼ他人です。変数名・関数名・コメントで「意図」を残します。
実践
Solve → Rewrite → Reuse
まず動かす。そのあと、読みやすく書き直す。繰り返す部分は関数化します。
🌟 小学生でもわかる！
コードは「料理のレシピ」に似ています。自分だけが今わかるメモではなく、明日の自分や友達が読んでも同じ料理を作れるレシピにするのが大事です。
②
🧱 PART IIIの全体構造
このパートは、パイプで処理の流れを読む → 関数で繰り返しを減らす → ベクトルとリストでRのデータ構造を理解する → 反復処理で複数入力を処理する、という順番です。

Chapter 14 — Pipes with magrittr / パイプ
①
🚰 Pipe %>% の目的：処理の流れを読みやすくする
Pipeは、複数の処理を 上から下へ、物語のように読む ための道具です。Rでは関数を入れ子にできますが、入れ子が深くなると「内側から外側へ」読まなければならず、人間には読みにくくなります。
R — Pipe story
# Before: inside-out reading bop( scoop( hop(foo_foo, through = forest), up = field_mice ), on = head ) # After: top-to-bottom reading foo_foo %>% hop(through = forest) %>% scoop(up = field_mice) %>% bop(on = head)
🛠 修正したミス
元テキストのpipe例では field_mouse と単数形になっている箇所がありますが、前後の説明では field_mice なので、このページでは field_mice に統一しました。
②
🧭 同じ処理を書く4つの方法
Chapter 14では、同じ処理を4つの書き方で比較します。ポイントは「コンピュータにとって正しいか」だけでなく、人間が読んだときに意図が見えるかです。
書き方コードの特徴良い点弱点
intermediate objectsfoo_foo_1, foo_foo_2 を作る途中結果を確認しやすい名前が増えすぎる。番号ミスが起きる
overwrite同じオブジェクトを上書きする短く書ける途中で失敗するとデバッグしにくい
composition関数を入れ子にする代入が不要内側から外側へ読むので読みにくい
pipex %>% f() %>% g()処理の順番が見えるpipeを知らない人には説明が必要
③
⚙️ Pipeの裏側：一時オブジェクトに入れて順番に処理する
%>% は魔法ではなく、左側の結果を次の関数の第1引数として渡す仕組みです。イメージとしては、一時的な . という箱に結果を入れて、次の処理へ渡す と考えるとわかりやすいです。
R — lexical transformation idea
my_pipe <- function(.) { . <- hop(., through = forest) . <- scoop(., up = field_mice) bop(., on = head) } my_pipe(foo_foo)
④
🚫 Pipeを使わない方がよい場面
Pipeは万能ではありません。短く、直線的に、1つの主役オブジェクトが変化する処理に向いています。逆に、複雑な依存関係や複数入力には不向きです。
Use pipe
1つのデータフレームに対して、filter → mutate → summarize のように順番に変換する。
Avoid pipe
10ステップ以上、途中結果を頻繁に確認したい、複数のデータを複雑に組み合わせる。
⚠ 特に注意：環境とlazy evaluation
assign(), get(), load() のように現在の環境を使う関数や、tryCatch(), suppressMessages() のように遅延評価に依存する関数は、pipeと相性が悪いことがあります。
⑤
🧰 magrittrの便利な派生pipe
通常の %>% 以外にも、magrittrには便利なpipeがあります。ただし、%<>% は代入が見えにくくなるので、学習段階では使いすぎ注意です。
Operator名前何をするか例
%>%pipe左の結果を右の関数へ渡すx %>% mean()
%T>%tee pipe右側を実行するが、左側をそのまま返すx %T>% plot() %>% str()
%$%exposition pipeデータフレーム内の列名を直接使えるmtcars %$% cor(disp, mpg)
%<>%assignment pipe結果を左辺に上書き代入するmtcars %<>% transform(...)

Q1
%>% の主な目的はどれか？
✅ 正解。pipeの目的は、処理の流れを人間が読みやすくすることです。
❌ pipeは速度改善やNA処理の道具ではありません。主目的は可読性です。
Q2
関数合成 bop(scoop(hop(x))) の弱点は？
✅ 正解。関数合成はコンピュータには正しいですが、人間は内側から読む必要があります。
❌ 弱点は「読む方向」です。処理の順番が直感的に見えません。
Q3
pipeを避けた方がよいケースはどれか？
✅ 正解。長すぎるpipeは、意味のある中間オブジェクトに分けた方が読みやすくデバッグしやすいです。
❌ pipeは短い直線的な処理に向いています。長すぎる場合は分けましょう。
Q4
%T>% の特徴はどれか？
✅ 正解。plot()など副作用だけ欲しい処理をpipeの途中に入れるとき便利です。
❌ %T>% はtee pipeです。右側を実行しても、返すのは左側です。
Q5
%$% が便利な場面は？
✅ 正解。例：mtcars %$% cor(disp, mpg) のように列名を直接使えます。
❌ %$% はデータフレームの列を「展開」して使うためのpipeです。

Chapter 15 — Functions / 関数
①
♻️ 関数を書く理由：copy-and-pasteを卒業する
同じコードを何度もコピーすると、変数名の変更忘れなどの小さなミスが起きます。Chapter 15の中心は、3回同じようなコードを書いたら関数化を考える という実践ルールです。
R — Copy-paste mistake
# df$b の分母で min(df$a) を使ってしまっている df$b <- (df$b - min(df$b, na.rm = TRUE)) / (max(df$b, na.rm = TRUE) - min(df$a, na.rm = TRUE)) # 関数化すれば、この種類のミスを減らせる rescale01 <- function(x) { rng <- range(x, na.rm = TRUE) (x - rng[1]) / (rng[2] - rng[1]) }
②
🧪 関数化の手順：動くコード → 一般化 → 関数化
良い関数は、いきなり頭の中で完成させるものではありません。まず具体例でコードを動かし、入力を一般名に置き換え、重複を整理してから関数にします。
Step 1
具体例で動かす
df$a など、実データで処理を確認する
Step 2
入力を一般名に
x <- df$a として処理の本質を見る
Step 3
関数に包む
function(x) { ... } で再利用可能にする
R — rescale01()
rescale01 <- function(x) { rng <- range(x, na.rm = TRUE) (x - rng[1]) / (rng[2] - rng[1]) } rescale01(c(-10, 0, 10)) #> [1] 0.0 0.5 1.0 rescale01(c(1, 2, 3, NA, 5)) #> [1] 0.00 0.25 0.50 NA 1.00
③
✍️ 人間が読める関数名・コメント・スタイル
関数はコンピュータのためだけではありません。人間が「何をする関数か」を理解できる名前にします。基本は、関数名は動詞、引数名は名詞です。複数語は snake_case で統一します。
悪い例良い例理由
f()rescale01()短すぎる名前は意味が伝わらない
my_awesome_function()impute_missing()何をするのかが名前からわかる
rowMaxes()row_maxes()命名スタイルを統一する
コメントは「what」より「why」
コードを読めば何をしているか分かるのが理想です。コメントでは「なぜこの方法を選んだのか」「他に試した方法はなぜ使わなかったのか」を残すと、未来の自分に役立ちます。
④
🔀 条件分岐：if, else, switch()
関数の中では、入力に応じて処理を変える必要があります。if は必ず 長さ1のTRUE/FALSE を期待します。複数条件が長くなる場合は、関数を分けるか switch() を検討します。
R — conditional execution
greeting <- function(time) { if (time < 12) { "Good morning" } else if (time < 18) { "Good afternoon" } else { "Good evening" } }
⑤
🎯 引数・戻り値・pipeable function
関数を使いやすくするには、引数名・デフォルト値・戻り値を明確にします。Rでは通常、最後に評価された値が戻り値になります。pipeで使える関数にしたい場合は、主役データを第1引数に置くのが基本です。
Transformation function
入力を受け取り、変換したオブジェクトを返す。pipeに向く。例：filter(), mutate()
Side-effect function
表示・保存・プロットなど、何かを実行することが主目的。戻り値は重要でない場合が多い。
🌟 小学生でもわかる！
関数は「自動販売機」です。お金や番号が 引数、出てくる飲み物が 戻り値。pipeでつなげるなら、飲み物が次の機械にそのまま渡せる形だと便利です。

Chapter 16 — Vectors / ベクトル
①
🧬 Rの中心：すべてはvectorから始まる
Chapter 16では、Rのデータ構造を学びます。Rで扱う多くのものはvectorに基づいています。大きく分けると、atomic vector と list があります。
Atomic vector
すべての要素が同じ型。例：logical, integer, double, character。
List
要素ごとに型が違ってもよい。データフレームやtibbleの土台にもなる。
R — vector basics
typeof(c(TRUE, FALSE)) #> [1] "logical" typeof(c(1L, 2L, 3L)) #> [1] "integer" typeof(c(1.5, 2.5)) #> [1] "double" typeof(c("a", "b")) #> [1] "character"
②
🔢 4つの重要なatomic vector
まずは4種類を確実に区別します。特に integer と double はどちらも数値に見えるので注意が必要です。
型例意味よくある用途
logicalTRUE, FALSE真偽値条件、filter、比較結果
integer1L, 2L整数カウント、index
double1.5, pi小数を含む数値測定値、平均、標準偏差
character"Bangkok"文字列名前、カテゴリ、ラベル
③
⚠️ Coercion / 型変換：混ぜると1つの型にそろえられる
Atomic vectorは1つの型しか持てません。違う型を混ぜると、Rが自動的に1つの型へ変換します。これは便利ですが、予想外の結果の原因にもなります。
R — coercion
str(c(FALSE, 1L, 2.5, "three")) #> chr [1:4] "FALSE" "1" "2.5" "three" as.numeric(c("1", "2", "oops")) #> Warning: NAs introduced by coercion #> [1] 1 2 NA
変換の強さのイメージ
logical → integer → double → character の順に、より多くの情報を表せる型へ寄せられます。文字列が混ざると、全部characterになりやすいです。
④
📏 Recycling rules / 長さの違うvectorをどう扱うか
Rでは、長さの短いvectorが長いvectorに合わせて繰り返されることがあります。これを recycling と呼びます。便利ですが、意図しない繰り返しに注意が必要です。
R — recycling
1:6 + 10 #> [1] 11 12 13 14 15 16 1:6 + c(10, 20) #> [1] 11 22 13 24 15 26 1:6 + c(10, 20, 30, 40) #> Warning: longer object length is not a multiple of shorter object length
⑤
🔎 Subsetting / 必要な要素を取り出す
vectorから要素を取り出す方法は複数あります。positive integer、negative integer、logical、character namesの違いを押さえると、データ操作のエラーが減ります。
方法例意味
positive integerx[c(1, 3)]1番目と3番目を残す
negative integerx[-2]2番目を除く
logicalx[x > 0]条件がTRUEのものだけ残す
characterx[c("a", "b")]名前で取り出す
⑥
🏷 Attributes, S3 class, augmented vectors
Rのvectorには属性（attributes）を付けられます。特に names, dim, class は重要です。factor、date、date-time、tibbleは、基本vectorにclassなどの属性を加えた augmented vectors として理解できます。
Factor
カテゴリ + levels
内部的にはintegerにlevels属性が付いたもの。順序やカテゴリ管理に使う。
Date / POSIXct
数値 + class
Dateは1970-01-01からの日数、POSIXctは秒数を表す。
Tibble
list + data.frame class
各列は同じ長さのvector。listを拡張した構造。
S3
classで振る舞いを変える
print()などの総称関数がclassに応じて動く。
🌟 小学生でもわかる！
vectorは「同じ種類の箱が横に並んだもの」。listは「いろいろな箱を入れられる大きな箱」。tibbleは「同じ長さの箱の列を、表みたいに並べたもの」です。

Chapter 17 Preview — Iteration / 反復処理
①
🔁 Functionsの次に必要になるもの：同じ処理を複数入力へ
関数は「繰り返されるコード」を1つにまとめます。しかし実際の分析では、同じ関数をたくさんの列・ファイル・モデル・グループに適用したい場面が出てきます。そこで必要になるのがiterationです。
Function
1つの処理を名前付きの道具にする。例：rescale01(x)
Iteration
その道具を複数の入力に何度も使う。例：すべての数値列をrescaleする。
R — iteration idea
# Concept only: detailed Chapter 17 text was not in the attachment cols <- list(a = df$a, b = df$b, c = df$c) for (name in names(cols)) { cols[[name]] <- rescale01(cols[[name]]) }
