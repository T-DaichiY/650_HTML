Unit 1 — Chapter 21 / R Markdown
①
📄 R Markdownの3要素
R Markdownは、prose（文章）・code（コード）・results（結果） を1つの文書に統合する仕組みです。分析レポート、共同作業、分析ノートの3用途に使えます。
YAML Header
--- で囲む
title, date, output, paramsなど
Code Chunks
```{r}
Rコードと出力を管理する単位
Markdown Text
#, **bold**
見出し、リスト、リンク、表など
SOURCE .Rmd
--- title: "Diamond sizes" output: html_document --- ```{r setup, include=FALSE} library(ggplot2) smaller <- diamonds |> filter(carat <= 2.5) ``` We have `r nrow(diamonds)` diamonds. ```{r, echo=FALSE} ggplot(smaller, aes(carat)) + geom_freqpoly(binwidth = 0.01) ```
RENDERED REPORT

Diamond sizes

We have 53,940 diamonds.

②
⚙️ Knitの仕組み：.Rmd → knitr → Markdown → pandoc → output
Knitを押すと、R Markdownはまず knitr に送られ、コードが実行されてMarkdownになります。その後 pandoc がHTML、PDF、Wordなどの完成文書に変換します。
.Rmdsource文章 + コード
knitrexecute chunksコード実行
.mdmarkdown結果入り中間ファイル
pandocfinal outputHTML/PDF/Word
R — render manually
> rmarkdown::render("diamond-sizes.Rmd") # Knit buttonと同じように、レポートを生成する
③
🧱 Chunk options：誰に何を見せるかを制御する
R Markdownで一番大事なのはchunk optionです。意思決定者向けならコードを隠し、分析者向けならコードと結果を見せます。
Option意味典型用途
eval = FALSEコードを実行しない例としてコードだけ見せる
include = FALSE実行するがコードも結果も非表示setup chunk
echo = FALSEコードを隠し、結果だけ見せる意思決定者向けレポート
message = FALSEメッセージを隠すlibrary読み込みの余計な表示を消す
warning = FALSE警告を隠す完成版で不要な警告を消す
cache = TRUE時間のかかる結果を保存重い処理の再計算を避ける
⚠ message/warningを隠しすぎない
完成版では便利ですが、デバッグ中に隠すと原因が見えません。最初は表示し、最終レポートで必要に応じて消すのが安全です。
④
🔁 Cacheとdependency：速くするが、危険もある
cache = TRUE は、重い計算を再利用できる便利な機能です。ただし、chunk内のコード変更は検知しても、外部ファイルや別chunkの変更を自動で完全には追跡しません。
Rmd — cache dependency
```{r raw_data} rawdata <- readr::read_csv("large_file.csv") ``` ```{r processed_data, cache=TRUE, dependson="raw_data"} processed_data <- rawdata |> filter(!is.na(import_var)) |> mutate(new_variable = x + y) ``` # 外部ファイル変更も検知したいなら cache.extra を使う ```{r raw_data, cache.extra=file.info("large_file.csv")}
⑤
🧩 Inline code と parameters
本文中に `r nrow(diamonds)` のように書くと、Knit時に計算結果が文章に入ります。さらに params を使うと、同じRmdからクラス別、支店別、国別などのレポートを量産できます。
Inline code
`r nrow(df)`
本文中の数値を自動更新
Parameterized report
params$my_class
同じtemplateで条件だけ変える
YAML — parameters
--- output: html_document params: my_class: "suv" --- # Fuel economy for `r params$my_class`s ```{r} class <- mpg |> filter(class == params$my_class) ggplot(class, aes(displ, hwy)) + geom_point() ```

Unit 2 — Chapter 22 / Graphics for Communication with ggplot2
①
📊 探索用グラフと説明用グラフは違う
探索用グラフは自分が素早く理解するためのものです。説明用グラフは、背景を知らない相手がすぐ理解できるように作ります。つまり、タイトル・ラベル・注釈・スケール・テーマ が重要になります。
Exploratory
自分用
変数名が短くても、自分は意味がわかる
Expository
他人用
何が重要かをグラフ自体が説明する
②
🏷️ labs(): タイトルは「何の図か」ではなく「何が分かったか」
良いタイトルは、plot typeの説明ではなく、主な発見を短く伝えます。軸ラベルには正式名称と単位を入れ、captionにはデータソースを書きます。
R — labels for communication
ggplot(mpg, aes(displ, hwy)) + geom_point(aes(color = class)) + geom_smooth(se = FALSE) + labs( title = "Fuel efficiency generally decreases with engine size", subtitle = "Sports cars are a notable exception because of their light weight", x = "Engine displacement (L)", y = "Highway fuel economy (mpg)", colour = "Car type", caption = "Data from fueleconomy.gov" )
③
✍️ Annotation：見てほしい点に直接ラベルを置く
凡例を読ませるより、グラフ上に直接ラベルを置いた方が早いことがあります。重なるラベルは ggrepel::geom_label_repel() が便利です。
関数役割使いどころ
geom_text()文字ラベル軽い注釈
geom_label()背景付きラベル読みやすくしたい時
ggrepel::geom_label_repel()重なり回避複数ラベルが密集するとき
geom_segment()矢印や線特定点に注意を向ける
geom_rect()範囲強調特定領域を囲む
④
🎚️ Scales：軸・色・凡例を調整する
scaleは、データ値を視覚要素に変換する仕組みです。breaks/labelsで軸の読みやすさを整え、log scaleやColorBrewer/viridisで伝わりやすさを上げます。
scale_y_continuous(breaks = seq(15, 40, by = 5))
軸の目盛り位置を自分で指定します。
labels = NULL
値を見せられない時や、絶対数よりパターンを見せたい時にラベルを消せます。
scale_x_log10() + scale_y_log10()
見た目はlog変換ですが、軸ラベルは元のスケールで読めます。
scale_color_brewer() / viridis::scale_fill_viridis()
色覚多様性にも配慮した色を使えます。
guides(color = guide_legend(...))
凡例の行数、点の大きさ、表示位置を調整できます。
⑤
🔍 Zoom / Theme / Saving
拡大には coord_cartesian() が安全です。データを捨てずに表示範囲だけを変えるため、smoothなどの計算結果が変わりにくいです。保存時は ggsave() またはR Markdownのfigure optionsを使います。
Zoom
coord_cartesian()
表示範囲だけを変える
Theme
theme_bw()
非データ部分の見た目
Save
ggsave()
幅・高さを指定して保存
Rmd — figure options
```{r, fig.width=6, fig.asp=0.618, out.width="70%", fig.align="center"} ggplot(mpg, aes(displ, hwy)) + geom_point() ```

Unit 3 — Chapter 23 / R Markdown Formats
①
📦 Output format はYAMLで決まる
R MarkdownはHTMLだけでなく、PDF、Word、Notebook、Slides、Dashboard、Website、Bookなど多くの出力形式を作れます。出力形式はYAML headerの output: で指定します。
YAML — multiple outputs
--- title: "Viridis Demo" output: html_document: toc: true toc_float: true pdf_document: default word_document: default ---

📄 Documents / Notebooks / Presentations
同じRmdでも、読み手と用途に合わせて出力形式を変えられます。意思決定者向けはHTML/PDF/Word、分析者同士の共有にはNotebookやGitHub document、発表にはpresentation形式が使えます。

Unit 4 — Chapter 24 / R Markdown Workflow
①
📓 Analysis notebook：何をしたかだけでなく、なぜしたかを残す
R Markdownは、consoleとscript editorをつなぐ分析ノートになります。chunk内で試し、うまくいったら文章で考えを残す。これにより、未来の自分や同僚が「なぜこの分析をしたのか」を理解できます。
Record
何をしたか
コードと結果を残す
Reflect
なぜしたか
仮説・判断・迷いを書く
Share
他人が理解する
チーム作業の文脈を残す
②
✅ 良いanalysis notebookのチェックリスト
分析ノートは、単なるコード置き場ではありません。目的、日付、失敗、データ修正の理由、再現性を記録します。
項目なぜ重要か
descriptive title何の分析かすぐ分かる
ISO dateYYYY-MM-DD で曖昧さをなくす
first paragraph分析の目的を最初に明記
dead ends失敗した分析を削除せず、なぜ失敗したか残す
data fixes as code元データを直接編集せず、修正理由をコードで記録
knit before leavingその日のうちに再現できるか確認
sessionInfo()使用package versionを後で確認できる
③
🧪 失敗を消さない：dead endも情報
時間をかけた分析がうまくいかなかった場合、削除すると将来また同じ道に迷います。短いメモで「何を試したか」「なぜダメだったか」を残します。
Notebook note — dead end example
## Attempt: model price using raw carat This did not work well because the relationship was strongly curved. The residual plot showed clear structure, so I switched to log2(price) and log2(carat) in the next section. ```{r raw-carat-model} mod_raw <- lm(price ~ carat, data = diamonds) ```
④
🔒 Reproducibility：来月・来年も動くようにする
長期的に再現したいなら、package versionを残す必要があります。厳密にはpackratやcheckpointのような環境固定が必要ですが、簡易的には sessionInfo() を最後に入れるだけでも大きく違います。
R — reproducibility footer
```{r session-info} sessionInfo() ```
⚠ Knitできないノートは、まだ完成していない
その日の終わりに必ずKnitして、エラーがないか確認します。エラーは翌日には文脈を忘れて修正が難しくなります。
